# Phase 6 → 2B coupling proof

**Decision:** **DECOUPLED.** Phase 6 and Phase 2B can run in either order, independently.

This document is the audit artefact for **Phase 0.3** of
`road-to-structural-optimization.md`. It enumerates every Phase-2B
target rule, records its references to the three Phase-6-owned rules
(`chat-history-cadence`, `chat-history-ownership`,
`chat-history-visibility`), and classifies each match as either a
**rule reference** (Phase-6 collision risk) or a **dispatcher
reference** (separate CLI infrastructure, not affected by Phase 6).

Source of truth for the file set: `.agent-src.uncompressed/rules/`.
The compressed mirror under `.agent-src/rules/` is regenerated by
`task sync` and matches by construction; the linter (0.3.3) checks
both surfaces to catch drift.

## Scope

**Targets:** 13 Phase-2B `type: auto` rules per the priority list at
`agents/roadmaps/archive/road-to-structural-optimization.md` § Phase 2 → 2B.

**Probes:**

1. Rule-name matches: literal occurrences of `chat-history-cadence`,
   `chat-history-ownership`, `chat-history-visibility`.
2. `load_context:` frontmatter entries pointing at any of the three
   rule files.
3. Body-link / explicit-cite forms (Markdown links, inline
   backtick references).

**Excluded from coupling:** the CLI dispatcher subcommand
`./agent-config chat-history:hook` and any other `chat-history:*`
command surface. Phase 6 reshapes the three **rule** files; the
dispatcher contract is owned by `scripts/agent_config_cli/` and
predates / outlives Phase 6. A reference to `chat-history:hook` from
a Phase-2B rule is therefore not a coupling event.

## Detail — `docs-sync` dispatcher mention

Line 88 of `.agent-src.uncompressed/rules/docs-sync.md` (and the
identical compressed copy at `.agent-src/rules/docs-sync.md`):

> `templates/consumer-settings/claude-settings.json` — Only when a
> **new platform event** is wired (e.g. first time a hook listens on
> `SessionEnd`). Existing hooks reuse the
> `./agent-config chat-history:hook --platform claude` dispatcher —
> no per-hook entry needed.

This is a **CLI dispatcher reference**, not a rule reference. Phase 6
restructures the three `chat-history-*` **rules** (per Phase 6 scope
in the parent roadmap, § Phase 6); the CLI dispatcher
(`scripts/agent_config_cli/`) is untouched by Phase 6. Sequencing is
unaffected.

## Outcome

Per Phase 0.3.2 case-table:

> **0 hits → decouple confirmed; Phase 6 ↔ Phase 2B can run in either order.**

The "0 hits" verdict applies on the rule-coupling probe (rule names,
`load_context:`, body link/cite). The single dispatcher mention in
`docs-sync` is excluded from the coupling probe per the scope
section above. The roadmap can therefore drop the **"Phase 6 must
ship a backward-compatible call-signature contract before Phase 2B
touches the coupled rule"** alternative — no coupled rule exists.

## Linter

`scripts/check_phase_coupling.py` re-runs the audit on every PR
that touches a Phase-2B rule and fails the build if a new
rule-name / `load_context:` / body-link reference is introduced.
The dispatcher exclusion is encoded in the linter's regex set so
the verdict stays stable across maintenance churn.

## Re-validation triggers

Re-run the audit (and the linter) when **any** of the following
land:

1. A new rule is added to the Phase-2B priority list.
2. A Phase-2B rule starts referencing chat-history rules during
   slimming (would be caught by 2A.4 / 2B.4 obligation diff first,
   linter is defence in depth).
3. Phase 6 expands scope from the three rule files to the dispatcher
   contract — would invalidate the dispatcher exclusion and require
   a fresh classification.
